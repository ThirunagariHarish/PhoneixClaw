# SemVer Rationale — v2.0.0

**Version:** 2.0.0  
**Previous Version:** 1.15.3  
**Bump Type:** MAJOR (breaking changes)  
**Rationale Date:** 2026-04-18

---

## Summary

The version bump from **1.15.3** to **2.0.0** is a **MAJOR** release per [Semantic Versioning 2.0.0](https://semver.org/spec/v2.0.0.html) due to multiple breaking changes in the public API, required fields, and removed functionality.

Per SemVer §8:

> Major version X (X.y.z | X > 0) MUST be incremented if any backwards incompatible changes are introduced to the public API. It MAY also include minor and patch level changes. Patch and minor version MUST be reset to 0 when major version is incremented.

This release introduces **four breaking changes** that justify the major bump:

1. **Removed OpenClaw Bridge Service** — Entire API surface removed
2. **Removed Alpaca broker adapter** — Core functionality removed
3. **`engine_type` required field** — API contract change
4. **Makefile targets removed** — Developer tooling breaking change

---

## Breaking Changes Detail

### 1. OpenClaw Bridge Service Removed

**Type:** Public API removal  
**Impact:** High — All `/api/v2/bridge/*` endpoints return 404  
**Justification:** Entire remote agent execution path removed. Any clients or integrations relying on OpenClaw endpoints will break.

**Before:**
```http
POST /api/v2/bridge/agents/create
GET /api/v2/bridge/agents/{id}/status
```

**After:**
```http
404 Not Found
```

**Migration Path:** Use local agent execution via Agent Gateway (`/api/v2/agents`).

**SemVer Rule:** Removal of public API endpoints is a breaking change (§8).

---

### 2. Alpaca Support Removed

**Type:** Core functionality removal  
**Impact:** Medium — Agents configured with Alpaca connectors will fail to execute trades  
**Justification:** `AlpacaBrokerAdapter` class deleted. No fallback or deprecation path provided.

**Before:**
```python
from shared.broker_adapters.alpaca import AlpacaBrokerAdapter
adapter = AlpacaBrokerAdapter(api_key="...", secret_key="...")
```

**After:**
```python
ModuleNotFoundError: No module named 'shared.broker_adapters.alpaca'
```

**Migration Path:** Switch to Robinhood MCP or IBKR Gateway adapters.

**SemVer Rule:** Removal of public classes or modules is a breaking change (§8).

---

### 3. `engine_type` Required Field

**Type:** API contract change  
**Impact:** High — All agent creation requests without `engine_type` field return 400 Bad Request  
**Justification:** Previously optional field is now mandatory. Clients must update request payloads.

**Before (v1.15.3):**
```json
POST /api/v2/agents
{
  "name": "My Agent",
  "type": "trading",
  "connector_id": "..."
}
```
**Response:** 200 OK (default `engine_type=legacy` inferred)

**After (v2.0.0):**
```json
POST /api/v2/agents
{
  "name": "My Agent",
  "type": "trading",
  "connector_id": "..."
}
```
**Response:** 400 Bad Request — `engine_type` field required

**Correct v2.0.0 Payload:**
```json
{
  "name": "My Agent",
  "type": "trading",
  "connector_id": "...",
  "engine_type": "pipeline"
}
```

**SemVer Rule:** Adding a required field to an API request schema is a breaking change (§8). Clients that do not provide `engine_type` will fail.

---

### 4. Makefile Targets Removed

**Type:** Developer tooling breaking change  
**Impact:** Low (developer-facing only) — CI/CD pipelines or scripts referencing removed targets will fail  
**Justification:** `run-bridge` and `test-bridge` targets deleted. No deprecation warning provided.

**Before:**
```bash
make run-bridge
make test-bridge
```

**After:**
```bash
make: *** No rule to make target 'run-bridge'. Stop.
make: *** No rule to make target 'test-bridge'. Stop.
```

**Migration Path:** Remove references to these targets from scripts and CI/CD pipelines.

**SemVer Rule:** While Makefile targets are not strictly "public API," they are part of the documented developer workflow (`CLAUDE.md` references `make go-live-regression` which included `test-bridge`). Removal without deprecation is breaking.

## Why Not a Minor Release?

Per SemVer §7:

> Minor version Y (x.Y.z | x > 0) MUST be incremented if new, backwards compatible functionality is introduced to the public API. It MUST be incremented if any public API functionality is marked as deprecated. It MAY include patch level changes. Patch version MUST be reset to 0 when minor version is incremented.

A minor release is only valid if **all changes are backwards compatible**. The four breaking changes listed above violate backwards compatibility:

- Clients cannot use OpenClaw endpoints in v2.0.0 (they could in v1.15.3).
- Clients cannot create agents without `engine_type` in v2.0.0 (they could in v1.15.3).
- Code cannot import `AlpacaBrokerAdapter` in v2.0.0 (it could in v1.15.3).
- Scripts cannot run `make test-bridge` in v2.0.0 (they could in v1.15.3).

Therefore, a **minor** bump (e.g., 1.16.0) is **not** appropriate.

---

## Why Not a Patch Release?

Per SemVer §6:

> Patch version Z (x.y.Z | x > 0) MUST be incremented if only backwards compatible bug fixes are introduced. A bug fix is defined as an internal change that fixes incorrect behavior.

This release includes multiple breaking changes, new features (pipeline engine, Grafana dashboards), and removed functionality. A **patch** bump (e.g., 1.15.4) is **not** appropriate.

---

## Added Features (Would Be Minor If Standalone)

If the breaking changes were not present, the following additions would justify a **minor** bump:

- Pipeline-based execution engine (new functionality, backwards compatible if `engine_type` defaulted)
- Robinhood MCP integration (new broker adapter)
- IBKR Gateway support (new broker adapter)
- Backtesting database persistence (new feature)
- Grafana dashboards (new observability tooling)
- 8 operational runbooks (new documentation)

However, because these features are bundled with breaking changes, the **major** bump takes precedence.

## Alternative Versioning Considered

### Option 1: Delay Breaking Changes (Not Chosen)
- Ship v1.16.0 with new features only.
- Deprecate OpenClaw and Alpaca in v1.16.0 with warnings.
- Remove deprecated features in v2.0.0 six months later.

**Why Not Chosen:** Business requirement to remove OpenClaw immediately (no ongoing maintenance cost). Delaying removal would require maintaining two execution paths (pipeline + OpenClaw) through multiple releases.

### Option 2: Micro-Version Bumps (Not Chosen)
- Ship v1.16.0 for pipeline engine.
- Ship v2.0.0 for OpenClaw removal.
- Ship v2.1.0 for `engine_type` requirement.

**Why Not Chosen:** All breaking changes are part of a single coordinated go-live phase. Shipping them separately would create partial upgrade states with unclear migration paths.

---

## Conclusion

The version bump from **1.15.3** to **2.0.0** is justified because:

1. **Four breaking changes** are introduced (API removal, required field, removed adapter, removed Makefile targets).
2. These changes violate backwards compatibility per SemVer §8.
3. Clients upgrading from v1.15.3 to v2.0.0 **must** modify their code or configurations (not a drop-in replacement).

The major bump signals to users that **manual migration steps are required** and that this is not a routine patch or feature release.

Per SemVer:

> MAJOR version when you make incompatible API changes

This release makes incompatible API changes. Therefore, **MAJOR** bump is required.
