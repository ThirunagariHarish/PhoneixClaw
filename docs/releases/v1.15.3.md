## Highlights

v1.15.3 makes live-agent chat sessions genuinely useful for portfolio monitoring.
Before this release, an agent in a chat session had no live Robinhood data — it could
only answer from static `agent_context.json` written at provision time. A user asking
"what are my current positions?" would receive a disclaimer that no live connection
existed, even though the agent held valid credentials.

This release wires two capabilities directly into every chat turn for agents with
status `RUNNING` or `APPROVED`:

1. **Live portfolio context** — real-time Robinhood data (positions, account value,
   buying power, cash) is fetched via `RobinhoodContextFetcher` and injected into
   `agent_context.json` before the Claude session starts.

2. **Read-only MCP tools** — the Robinhood MCP server is wired into the chat
   workdir, giving the agent eight read-only tools it can call directly during the
   conversation. Order-placement tools are explicitly withheld.

Additionally, two correctness fixes land in `agent_gateway.py`: the
`_ensure_system_agent` INSERT now sets `created_at`/`updated_at`, and system-agent
display names are canonicalised via a lookup dict instead of a fragile string transform.

## What's New

### Live Portfolio Context Injection (`chat_responder.py`)

For every chat turn where the agent is live (`RUNNING` / `APPROVED`) and holds
Robinhood credentials, `respond_to_chat()` now calls
`RobinhoodContextFetcher.fetch(agent_id)` before building the workdir.  The result is
serialised to `live_portfolio` inside `agent_context.json`:

```json
{
  "positions": [...],
  "account_value": 12345.67,
  "buying_power": 3200.00,
  "cash": 500.00,
  "last_updated_at": "2026-04-08T12:34:56+00:00"
}
```

If the fetch raises an exception, a stub with an `"error"` key is written instead.
The prompt instructs the agent to surface that error message — never to silently claim
it has no connection.

### Read-Only MCP Tools in Chat (`chat_responder.py`)

When live credentials are present and `robinhood_mcp.py` exists in the template
directory, the chat workdir receives:

- `.claude/settings.json` written via the existing `_write_claude_settings()` helper
  (paper_mode=**false** for live agents, so real API calls are permitted for reads).
- `tools/robinhood_mcp.py` copied from the live-trader-v1 template.
- An expanded `allowed_tools` list passed to `ClaudeAgentOptions`:

```
["Bash", "Read",
 "mcp__robinhood__robinhood_login",
 "mcp__robinhood__get_positions",
 "mcp__robinhood__get_account",
 "mcp__robinhood__get_quote",
 "mcp__robinhood__get_account_snapshot",
 "mcp__robinhood__get_nbbo",
 "mcp__robinhood__get_watchlist",
 "mcp__robinhood__get_order_status"]
```

The prompt gains two new instructional sections when these features are active,
directing the agent to call `robinhood_login` first, then the relevant read tool,
and never place orders via chat.

### System-Agent Name Canonicalisation (`agent_gateway.py`)

A new module-level constant `_SYSTEM_AGENT_NAMES` maps `agent_type` string keys to
their canonical display names:

```python
_SYSTEM_AGENT_NAMES = {
    "eod_analysis":  "EOD Analysis Agent",
    "daily_summary": "Daily Summary Agent",
    "trade_feedback": "Trade Feedback Agent",
}
```

`_spawn_one_shot_agent` now looks up the name from this dict (with the old
`.replace("_", " ").title() + " Agent"` as fallback), ensuring names in the DB
exactly match the seed list in `main.py`.

## Bug Fixes

### `_ensure_system_agent` Missing Timestamps (`agent_gateway.py`)

The idempotent `INSERT … ON CONFLICT (id) DO NOTHING` that guards system-agent rows
omitted `created_at` and `updated_at` from the column list. On databases where those
columns carry no server-side `DEFAULT`, this produced a `NOT NULL` constraint
violation. The INSERT now supplies `NOW(), NOW()` for both columns.

### Import Hygiene (`agent_gateway.py`, `chat_responder.py`, tests)

- `sys` is now imported at module level in `agent_gateway.py` (was deferred as
  `import sys as _sys` inside `_write_claude_settings`).
- All `from claude_agent_sdk import ...` statements sorted alphabetically
  (`ClaudeAgentOptions, query`) across six call sites.
- `from shared.triggers import ...` sorted (`Trigger, TriggerType, get_bus`).
- `from sqlalchemy import ...` sorted in `chat_responder.py` (`desc, select`).
- Import order corrected in `test_agent_gateway_error_path.py` (ruff I001).

| Test | What it verifies |

**Behaviour change:** `_write_claude_settings(work_dir, rh_creds={}, paper_mode=True)`

entry with `PAPER_MODE=true` and no real credentials. This enables paper-mode agents
to exercise the MCP tool surface safely. `test_write_claude_settings_without_credentials`
has been updated to reflect the new expectation.

## Rollback Plan

### Quick rollback (< 3 minutes)

```bash
# 1. Return to the last clean v1.15.2 state
git checkout 2ac7ad1

# 2. Rebuild and restart the API service
docker compose up -d --build phoenix-api
```

### What rollback means

- Live portfolio data will no longer be injected into chat sessions.
- Read-only MCP tools will no longer be available in chat.
- Agents will revert to static-context chat responses.
- The `_ensure_system_agent` `NOT NULL` fix is also reverted; if your DB schema
  enforces `NOT NULL` on `created_at`/`updated_at` with no server-side default,
  system-agent creation (Morning Briefing, EOD, etc.) may fail again.
  In that case stay on `2ac7ad1` and cherry-pick only the timestamp fix.

> No data is at risk: this release writes no new DB tables and no migrations.
> Chat workdirs written during v1.15.3 are inert on the older codebase and can
> be left in place or pruned with `rm -rf agents/chat_sessions/`.
