# Phoenix Trade Bot v0.3.0 — Release Notes

**Release date:** 2026-04-08
**Type:** Minor — bug-fix + new paper-trading infrastructure
**Base:** v0.2.0 (Polymarket Tab)
**Commits:** `0b8433f` → `de8eec1` (7 commits on `main`)

---

## Highlights

This release resolves a **P0 signal starvation bug** that caused all paper-trading
agents to silently receive zero signals in production, a set of **agent creation
wizard regressions** (wrong final status, race condition on backtest rows,
non-trading types blocked), and **10 pipeline hardening items** surfaced during
end-to-end testing. It also ships new paper-trading infrastructure (dedicated
`CLAUDE.md` template, `log_paper_trade.py`, cursor persistence) and a new
`error_message` field on the `AgentResponse` API surface.

---

## Breaking changes

**None.** All API, schema, and template changes are backwards-compatible.
The new `error_message` field on `AgentResponse` is nullable and defaults to `null`.

---

## Migration steps

1. **Run the Alembic migration** before deploying the new image:
   ```bash
   alembic upgrade head
   ```
   Migration `032_agents_tab_fix.py` adds two nullable/defaulted columns:
   - `agents.error_message TEXT NULL`
   - `agent_sessions.trading_mode VARCHAR(20) DEFAULT 'live'`

   The migration is idempotent (safe to re-run) and takes < 1 second on
   typical table sizes.

2. **No other steps required.** No data backfill, no index rebuilds, no
   config file changes.

## What changed

### P0: Live signal pipeline restored

`live_pipeline.py` was reading from `discord_listener.get_signal_queue()` — an
in-process queue that is always empty in production (populated only by an
in-process thread, never by the separate Claude Code agent session). Every
paper-trading agent launched since the Redis Streams migration received zero
signals. Fixed: pipeline now reads from Redis via `_redis_signal_stream()`,
an async generator consuming `stream:channel:{connector_id}`.

The Discord Redis consumer (`discord_redis_consumer.py`) had three additional
P0 issues fixed:
- Was reading from stream key `channel_id` (Discord numeric ID) instead of
  `connector_id` (DB UUID). Ingestion writes to `stream:channel:{connector_id}`,
  so consumer never matched.
- Had a 30-second hard deadline (`--max-seconds 30`) that caused it to stop
  listening after half a minute.
- Started at cursor `$` (tail only), missing all backlogged messages; now
  starts at `0-0` when no cursor file exists, and persists last-read position
  to `stream_cursor.json`.

### Agent creation wizard

- Trend, sentiment, and news agent types can now complete the creation wizard
  without selecting a connector — they were incorrectly blocked by the trading-
  type connector guard.
- Agents that fail backtesting now surface `status=ERROR` with a red error
  banner on the dashboard card, rather than silently reverting to `CREATED`.
- Backtest race condition eliminated: row shows `PENDING` until the async task
  actually begins, then `RUNNING` → `COMPLETED/FAILED`.

### Paper trading safety

- Paper agents now use `CLAUDE.md.paper.jinja2` with an explicit prohibitions
  block — no Robinhood tool calls, no real order placement.
- Broker credentials are withheld from paper agents at config generation time.
- Paper agent status is preserved as `PAPER` through `create_analyst()` (was
  incorrectly overwritten to `RUNNING`).
- `live_pipeline.py` EXECUTE stdout gated on `not paper_mode` — paper agents
  can no longer accidentally emit live execution events.

## Rollback

### Automated rollback
```bash
alembic downgrade 031
```
Migration `032` downgrade drops `agents.error_message` and
`agent_sessions.trading_mode`. No user data is lost (both columns are
nullable / have a server default).

Then redeploy the previous image (`v0.2.0`).

### What rollback does NOT affect
- All Polymarket Tab (v0.2.0) data and functionality — untouched by this patch.
- Agent history, sessions, connector configs, backtest records — all preserved.
