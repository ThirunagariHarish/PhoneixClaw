# Analyst Agent — Product Requirements Document (PRD)

## 1. Executive Summary

The Analyst Agent feature introduces AI-powered trading agents that emulate real-world professional traders within Phoenix Trade Bot. Each analyst agent is imbued with a unique persona, such as "Aggressive Momentum Trader" or "Options Flow Specialist," and leverages a curated set of trader tools to analyze market signals, perform technical and sentiment analysis, and emit structured trade signals. This system aims to bridge the gap between algorithmic automation and human-like trading expertise, providing users with transparent, explainable, and persona-driven trade recommendations.

By orchestrating a workflow that mirrors the decision-making process of seasoned traders, Analyst Agents will process Discord signals, market data, and sentiment analysis in a repeatable, auditable manner. The output is a structured TradeSignal record, complete with reasoning and confidence scores, visible in the dashboard and ready for paper or live execution. This approach enhances trust, interpretability, and user engagement, while enabling rapid iteration on trading strategies and personas.

The Analyst Agent system is designed for extensibility, supporting new personas, tools, and workflows as the platform evolves. It leverages existing Phoenix Trade Bot assets—such as the Agent and TradeSignal models, Claude SDK session management, and broker adapters—while introducing new data fields, endpoints, and UI components to support the analyst paradigm. The phased rollout ensures robust delivery, from persona-driven signal generation to full live trading integration.

## 2. Problem Statement

Current algorithmic agents in Phoenix Trade Bot lack the transparency, explainability, and human-like reasoning that professional traders provide. Users want to understand not just what trades are recommended, but why—complete with context, tool usage, and persona-driven logic. There is a need for agents that can synthesize multiple sources (Discord signals, technicals, sentiment, options flow) and present actionable, auditable trade signals with clear reasoning, confidence, and risk management parameters.

## 3. User Personas
- **Retail Traders**: Seeking actionable, explainable trade ideas and transparency into agent logic.
- **Professional Traders**: Wanting to test, tune, and monitor AI agents that mimic real-world trading workflows.
- **Quant Developers**: Extending and customizing analyst personas, tools, and workflows.
- **Product Managers/Analysts**: Monitoring agent performance, signal quality, and user engagement.

## 4. Goals & Non-Goals

**Goals:**
- Enable creation of analyst agents with selectable personas and orchestrated workflows.
- Provide structured, explainable trade signals with full reasoning and tool traceability.
- Support paper and live trade submission via broker adapters.
- Expose signals and agent activity in the dashboard for transparency and monitoring.

**Non-Goals:**
- Building a generic LLM chat agent (focus is on structured, persona-driven workflows).
- Supporting non-equity asset classes (crypto, forex) in initial release.
- Real-time high-frequency trading (focus is on explainable, auditable workflows).

## 5. Epics & User Stories

### Epic 1: Analyst Agent Creation & Persona System
- As a user, I can create a new analyst agent and select from 6 personas.
- As a user, I can view persona descriptions and select the one that fits my strategy.
- As a developer, I can extend the persona library with new personas.

**Gherkin Acceptance Criteria:**
```
Given I am on the agent creation wizard
When I reach the persona selection step
Then I see 6 persona options with descriptions
And I can select one before proceeding
```

### Epic 2: Analyst Tool Manifest Implementation
- As an analyst agent, I can call trader tools (chart analysis, options flow, dark pool, sentiment, etc.) in a defined order.
- As a developer, I can add new tools to the manifest and configure tool weights per persona.

### Epic 3: Orchestrated Trading Workflow
- As an analyst agent, I follow a repeatable workflow: pre-market scan, signal intake, technical confirmation, sentiment check, scoring, decision, handoff.
- As a user, I can view the workflow steps and reasoning for each signal.

**Gherkin Acceptance Criteria:**
```
Given an analyst agent is running
When a new Discord signal is ingested
Then the agent processes it through all workflow steps in order
And emits a TradeSignal with full reasoning
```

### Epic 4: Trade Signal Output & Storage
- As an analyst agent, I emit structured TradeSignal records with persona, tool usage, confidence, and reasoning.
- As a user, I can view all signals generated by an analyst agent.

**Gherkin Acceptance Criteria:**
```
Given an analyst agent emits a signal
When I view the agent's detail page
Then I see the signal with ticker, direction, confidence, and reasoning
```

### Epic 5: Dashboard Signal Visibility
- As a user, I can see a "Signals" tab in the agent detail view.
- As a user, I can filter, sort, and expand signal cards to view reasoning and tool usage.

**Gherkin Acceptance Criteria:**
```
Given I am viewing an analyst agent's detail page
When I click the "Signals" tab
Then I see a list of recent signals with key details and expandable reasoning
```

### Epic 6: Paper & Live Trade Submission
- As an analyst agent, I can submit paper trades via Alpaca.
- As a user, I can view trade execution status and outcomes.
- As a developer, I can enable live trading via IBKR/Alpaca in Phase 3.

**Gherkin Acceptance Criteria:**
```
Given an analyst agent emits an EXECUTE signal
When paper trading is enabled
Then the agent submits a paper trade via Alpaca
And the trade status is updated in the dashboard
```

## 6. Functional Requirements

- **FR-001**: Add 'analyst' to Agent.type enum.
- **FR-002**: Extend agent creation wizard with persona picker (6 personas).
- **FR-003**: Store persona, skills, tool_weights, workflow_config in Agent.config JSONB.
- **FR-004**: Implement 10 analyst tools as callable functions/services.
- **FR-005**: Define orchestrated workflow for analyst agents (not random tool calls).
- **FR-006**: Analyst agents must process Discord signals as input.
- **FR-007**: Analyst agents must call tools in defined workflow order.
- **FR-008**: Analyst agents must emit TradeSignal records with persona, tool usage, confidence, reasoning.
- **FR-009**: Add analyst_persona, tool_signals_used, risk_reward_ratio, take_profit to TradeSignal model.
- **FR-010**: Store all emitted signals in DB and expose via API.
- **FR-011**: Implement POST /api/v2/agents/{id}/analyst/run endpoint.
- **FR-012**: Implement GET /api/v2/agents/{id}/signals endpoint.
- **FR-013**: Implement GET /api/v2/signals endpoint with filters.
- **FR-014**: Extend POST /api/v2/agents to accept analyst type + persona.
- **FR-015**: Add "Analyst" badge and last signal time to agent card in dashboard.
- **FR-016**: Add "Signals" tab to agent detail view.
- **FR-017**: Display signal cards with ticker, direction, confidence, reasoning.
- **FR-018**: Reasoning must be expandable/collapsible in UI.
- **FR-019**: Implement scheduler to run analyst agents every 15 min during market hours (Phase 2).
- **FR-020**: Implement paper trade submission via Alpaca (Phase 2).
- **FR-021**: Implement live trade submission via IBKR/Alpaca (Phase 3).
- **FR-022**: Support filtering and sorting signals in dashboard.
- **FR-023**: Expose tool usage and workflow steps in signal detail.
- **FR-024**: Support persona library extensibility (add/edit personas).
- **FR-025**: Log all tool calls and workflow steps for auditability.
- **FR-026**: Support agent state transitions (CREATED → APPROVED → EXECUTING → IDLE → DISABLED).
- **FR-027**: Integrate with existing agent_gateway.py for Claude SDK sessions.
- **FR-028**: Support multi-tenant operation (per-user agents, signals).
- **FR-029**: Provide API docs for all new/extended endpoints.
- **FR-030**: Ensure all new features are covered by unit and integration tests.

## 7. Non-Functional Requirements

- **NFR-001**: All analyst agent actions must be auditable (logs, tool calls, decisions).
- **NFR-002**: System must support at least 100 concurrent analyst agents.
- **NFR-003**: Signal generation latency < 10 seconds per agent per run.
- **NFR-004**: Dashboard must update signals in near real-time (<5s lag).
- **NFR-005**: All API endpoints must be authenticated and RBAC-enforced.
- **NFR-006**: Data model changes must be backward compatible.
- **NFR-007**: System must support phased rollout (feature flags per phase).
- **NFR-008**: All new code must pass existing lint/typecheck/test gates.
- **NFR-009**: UI must be responsive and accessible (WCAG 2.1 AA).
- **NFR-010**: All user-facing text must be localizable.

## 8. Data Model Changes

- **Agent.type**: Add 'analyst' to enum.
- **Agent.config** (JSONB): Add fields — persona (string), skills (list), tool_weights (dict), workflow_config (dict).
- **TradeSignal**: Add fields — analyst_persona (string), tool_signals_used (JSONB), risk_reward_ratio (float), take_profit (float).
- **New Table (optional)**: AnalystPersona — id, name, description, default_tool_weights, workflow_template.

## 9. API Contract

- **POST /api/v2/agents/{id}/analyst/run** — Trigger a manual analyst run for a given agent.
- **GET /api/v2/agents/{id}/signals** — Get all signals generated by a specific analyst agent.
- **GET /api/v2/signals** — Get all signals across analyst agents (supports filters: persona, ticker, date, confidence, status).
- **POST /api/v2/agents** (extend existing) — Accepts 'analyst' type and persona in payload.

## 10. UI Changes

- **Wizard Step 4**: Persona picker (6 personas, each with icon and description).
- **Agent Card**: Show "Analyst" badge and last signal time.
- **Agent Detail View**: New "Signals" tab listing recent signals.
- **Signal Card**: Show ticker, direction badge, confidence meter, expandable reasoning, tool usage summary.

## 11. Phased Delivery Plan

**Phase 1:**
- Add 'analyst' agent type, persona picker, tool manifest, orchestrated workflow, emit_trade_signal to DB, API endpoints, dashboard signal tab (read-only).
- **Dependencies:** Agent model, TradeSignal model, agent_gateway.py, dashboard UI.

**Phase 2:**
- Scheduler for periodic runs, paper trading integration, dashboard signal filtering/sorting, tool call logging, persona library extensibility.
- **Dependencies:** Broker adapters, scheduler infra, dashboard enhancements.

**Phase 3:**
- Live trading integration (IBKR/Alpaca), advanced persona tuning, pattern detection, multi-tenant scaling, localization.
- **Dependencies:** Broker live trading, advanced ML tools, i18n infra.

## 12. Success Metrics

- Signals generated per day per agent
- Signal confidence vs. realized trade outcome (accuracy)
- Paper trade win rate (%)
- User engagement: signals viewed, expanded, filtered
- Time to signal (latency)
- Persona adoption rates
- System uptime and error rates

## 13. Risks & Open Questions

- Ensuring persona behaviors are distinct and not just prompt engineering
- Tool reliability and data freshness (Discord, options flow, news sentiment)
- Avoiding overfitting or hindsight bias in scoring/trade decisions
- Scaling Claude SDK sessions for many concurrent agents
- Handling conflicting signals or tool outputs
- User trust in AI-generated signals (explainability, auditability)
- How to best tune personas for real-world performance
- Regulatory/compliance implications of live trading automation
