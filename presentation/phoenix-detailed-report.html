01 Key Idea — Vision & Approach

The Problem

The current landscape of automated trading is fundamentally broken. Traditional trading bots fall into two categories — both deeply flawed:

Hard-Coded Rule Bots

Rigid if/else logic that cannot adapt to changing market conditions. Requires constant manual updates. Breaks silently when market regimes shift.

Raw LLM Trading Bots

Newer AI bots let large language models make discretionary trading decisions. Unpredictable, untested, and dangerous — hallucinations become real losses.

Both approaches share critical weaknesses: no backtesting validation, no risk management layers, single points of failure, and no self-improvement capabilities. They are fundamentally gambling with code instead of trading with discipline.

The Solution: LLMs as Strategy Engineers

Core Principle: In ProjectPhoenix, LLMs are strategy engineers, not discretionary traders. Every AI agent must prove itself through rigorous backtesting before being allowed to trade with real capital.

ProjectPhoenix is an autonomous multi-agent trading platform that operates like a full trading firm staffed by specialized AI agents. The platform uses a custom web dashboard as a control plane to orchestrate multiple OpenClaw AI instances, each running specialized agents for different functions.

Design Principles

  • Intelligence in OpenClaw: The dashboard is control and display only. All reasoning, strategy evaluation, and trading logic runs inside OpenClaw AI agents.
  • Queue-Based Execution: Agents never call broker APIs directly. Trade intents are submitted to a queue and pass through validation and risk checks at every layer before execution.
  • Paired Agent Architecture: Every trading agent is automatically paired with a monitoring agent responsible for exits, stop-losses, take-profits, and position management.
  • Skill Reuse: Skills are authored once in a central repository and synced to all OpenClaw instances via MinIO — ensuring consistency and rapid capability expansion.
  • Fail-Safe Defaults: Circuit breakers, hard stop-losses, daily loss limits, and a global kill switch that can close all positions instantly.
  • Idempotent Operations: Every trade intent has a unique ID. The execution service deduplicates to prevent double-orders.
  • Secrets Never Leave Backend: API keys are encrypted with Fernet. The frontend sees masked previews only.
  • Self-Improving System: A Dev Agent with reinforcement learning monitors performance, detects issues, tunes parameters, and evolves strategies continuously.

The Trading Firm Analogy

Think of ProjectPhoenix not as a trading bot, but as an entire trading firm staffed by specialized AI agents:

  • Research Analysts — ingest and analyze signals from Discord, Twitter/X, Reddit, Unusual Whales, and news APIs
  • Strategy Engineers — design, backtest, and optimize trading strategies using LLM reasoning
  • Traders — execute approved strategies in live markets through queue-based pipelines
  • Risk Managers — monitor every position, enforce limits, and trigger circuit breakers when needed
  • Dev Agent (CTO) — monitors the entire operation, fixes bugs, tunes parameters, and improves strategies through reinforcement learning

03 Architecture — Three-Plane Design

Architecture Overview

ProjectPhoenix uses a three-plane architecture that separates concerns between control, execution, and shared services:

Control Plane

The control plane is the user-facing layer hosted on a central Coolify server. It provides the dashboard interface, API, database, and orchestration:

  • React Dashboard (Vite 5 + TypeScript + Tailwind + Radix UI) — 25+ pages covering agents, trades, positions, strategies, skills, market data, analytics, and administration
  • FastAPI Backend (Python 3.12 + Uvicorn) — 30+ route groups with JWT auth, MFA, RBAC (5 roles, 20 permissions)
  • PostgreSQL 16 + TimescaleDB — 18 relational tables and 3 hypertables for time-series data
  • Redis 7 — API cache, sessions, BullMQ job queue, Streams event bus, rate limiting, circuit breaker state
  • Orchestrator Worker — BullMQ-powered event processor that dispatches between services
  • WebSocket Gateway — Real-time push for trades, positions, agents, heartbeats, and alerts

Execution Plane

The execution plane runs on separate VPS nodes, each hosting an OpenClaw AI instance with a Bridge API sidecar:

  • Instance A — Strategy Lab: Agents that design, prototype, and backtest new trading strategies
  • Instance B — Data & Research: Agents focused on data ingestion, signal analysis, and market research
  • Instance C — Risk & Promote: Agents that evaluate strategies for promotion and manage risk assessment
  • Instance D — Live Trading: Approved agents executing trades in live markets with paired monitoring agents

Each instance runs a Bridge Service sidecar (port 18800) that exposes REST endpoints for agent CRUD, heartbeat, skill sync, and metrics. The bridge authenticates via shared token.

04 End-to-End Flow — Signal to Execution

Complete Trade Pipeline

This is the step-by-step journey of how a signal becomes a live trade in ProjectPhoenix:

1
Signal Arrives — A Discord message mentioning a ticker, a tweet from a trader, an Unusual Whales options flow alert, or a Reddit post is detected by the appropriate connector.
2
Ingestion & Normalization — The Connector Manager receives the raw signal, extracts tickers using NLP (BERT entity extraction), analyzes sentiment, normalizes the data into a standard format, and publishes to the Event Bus via stream:connector-events.
3
Agent Evaluation — A Strategy Agent in the appropriate OpenClaw instance receives the signal. It applies its backtested strategy, uses LLM reasoning to evaluate the opportunity, checks current market conditions, and decides whether to act.
4
Trade Intent Created — If the signal passes evaluation, the agent creates a trade intent specifying ticker, direction (buy/sell), quantity, order type, and risk parameters. This is posted to stream:trade-intents.
5
3-Layer Risk Validation — The trade intent passes through three risk layers:
  • Agent Risk: Per-agent position limits, daily loss limits, ticker concentration
  • Execution Risk: Per-trade validation, deduplication (unique trade IDs), order sanity checks
  • Global Risk: Portfolio-level circuit breaker, sector concentration, overall exposure limits
Any layer can reject the trade intent.
6
Broker Execution — The Execution Service places the order via the appropriate broker API (Alpaca, IBKR, Robinhood, or Tradier). The operation is idempotent — each trade intent has a unique ID to prevent double-orders.
7
Position Created — Once the order is filled, a position record is created in the database. A Monitoring Agent is automatically paired with the position to manage exits.
8
Real-Time Dashboard Updates — Position updates, P&L changes, and status updates are pushed to the dashboard via WebSocket in real-time through stream:position-updates.
9
Position Monitoring — The paired Monitoring Agent continuously tracks price, checks stop-loss levels, evaluates take-profit targets, and monitors exit conditions. It can close the position independently when criteria are met.
10
Dev Agent Learning Loop — After the trade closes, the Dev Agent with reinforcement learning analyzes the outcome, adjusts strategy parameters, re-backtests the improved strategy, and promotes it if it passes the gate criteria again.

Agent Lifecycle State Machine

Every agent follows a strict lifecycle with promotion gates and demotion triggers:

Created → Backtesting (50+ signals) → Paper Trading → Live Trading → RL Improvement

Promotion Gates

  • 50+ signals evaluated in backtest
  • Win rate ≥ 45%
  • Profit factor ≥ 1.2
  • Sharpe ratio ≥ 0.5
  • Max drawdown ≤ 30%

Demotion Triggers

  • Daily loss limit breached (-5%)
  • Error rate > 5% in 5 minutes
  • Agent heartbeat lost for 2+ minutes
  • Circuit breaker tripped
  • Manual kill switch activated

Self-Improvement Loop

The Dev Agent operates as the platform's CTO, continuously improving the system:

  1. Monitor: Watches all agent performance metrics, trade outcomes, and error rates in real-time
  2. Detect: Identifies underperforming agents, strategy drift, and system errors using statistical analysis
  3. Fix & Tune: Adjusts strategy parameters, fixes code issues, and optimizes agent configurations
  4. Re-Backtest: Runs the improved strategy through the full backtesting engine to validate changes
  5. Re-Deploy: If the improved strategy passes all gate criteria, promotes it back to live trading

This creates a continuous improvement cycle where agents evolve and adapt to changing market conditions autonomously.
